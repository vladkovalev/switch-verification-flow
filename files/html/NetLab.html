NetLab (GNS3)

NetLab is the front-end lab for Spectrum-2/3/4 development. It uses GNS3 to orchestrate network topologies (links, switches, node lifecycle) and runs the control-plane in a QEMU VM, while a separate simulation node models the ASIC/platform world. The key output is a replayable golden stimulus trace (transactions.parquet) that can be replayed in deeper debug systems.

Orchestration: GNS3
Compute: QEMU/KVM (CPU VM)
Model: SystemC LT/AT (SIM node)
Ports: mgmt0 + swp1..8
PCIe v1: ivshmem transport
Trace: Parquet ZSTD (128MB groups)

What GNS3 adds

  • Visual topology + lifecycle control
  • Internal virtual switching (bridges/OVS)
  • Shareable lab projects

What NetLab produces

  • transactions.parquet (golden stimulus)
  • run.json manifest + logs
  • Optional semantic traces (RTEdbg)

What NetLab avoids

  • Always-on waveform capture
  • Immediate dependency on emulation
  • Full cycle-accurate signoff
Tick/Tack packaging: Start with tick (two visible nodes: CPU + SIM) for modular development. When stable, package as tack (one wrapper node) without changing external ports or debug entrypoints.

Tick vs Tack (how it appears in GNS3)

Same ports + same debug entrypoints; different packaging
Guideline: develop and evolve functionality in tick. When interfaces and operations stabilize, ship tack to simplify consumption.

How to use (operator workflow)

Fast reproduction → capture trace → replay downstream
  1. Build topology in GNS3: management switch + dataplane switch(es) + nodes (CPU, SIM, clients).
  2. Start SIM node first (Debug Portal gRPC server, trace writer ready).
  3. Start CPU node, run the probe tool to exercise MMIO + MSI-X init path.
  4. Capture trace via Debug Portal (StartTrace / StopTrace) → transactions.parquet.
  5. Escalate only if needed: replay the same trace in DebugLab-Lite / DebugLab-Full.
# Example (conceptual)
TraceService.StartTrace(output_dir="run_001/",
  trace_buses=["PCIE_MMIO","MSIX"],
  parquet_compression="ZSTD",
  parquet_row_group_size_mb=128)

# Run probe/regression in tick-switch-cpu...

TraceService.StopTrace()
Best practice: treat NetLab traces as “portable bug capsules”. A single captured run should be enough to drive deeper waveform-level debugging later.
