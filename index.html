Overview

We build a tiered verification and debug ecosystem for a PCIe-attached device/ASIC with an x86_64 Intel/AMD management CPU: NetLab for topology/integration and fast reproduction, DebugLab-Lite for open-source waveform debug (Verilator + RTEdbg), and DebugLab-Full for the internal fidelity ladder ending with emulation as a last resort.

Debug Portal: gRPC
Golden stimulus: transactions.parquet
Compression: ZSTD
Row groups: 128MB
corr_id: globally unique per run
PCIe v1: MMIO + MSI-X (DMA optional)
Core idea: capture once (in NetLab), replay everywhere. Compare traces by seq + corr_id. Escalate fidelity only when required, and capture waveforms only around the first divergence.

Goal & why split the system

Goal

  • Fast iteration in simulation-first environments
  • Replayable evidence across all stages
  • Emulation only as a last resort

Why split

  • NetLab scales to many nodes and scenarios
  • Lite provides waveforms with open tools
  • Full provides the internal fidelity ladder

What makes it work

  • gRPC Debug Portal (uniform control plane)
  • Parquet golden stimulus (uniform evidence)
  • corr_id anchors (uniform correlation)
EDA vendor neutrality: DebugLab-Full is designed to integrate with internal commercial stacks as needed. Current deployments may use Cadence, Synopsys, and/or Siemens tooling (deployment-dependent).

NetLab (GNS3)

NetLab uses GNS3 to orchestrate network topologies (links, switches, node lifecycle) and runs the control-plane in a QEMU VM, while a separate simulation node models the ASIC/platform world. The key output is a replayable golden stimulus trace (transactions.parquet) that can be replayed in deeper debug systems.

Orchestration: GNS3
Compute: QEMU/KVM (CPU VM)
Model: SystemC LT/AT (SIM node)
Ports: mgmt0 + swp1..8
PCIe v1: ivshmem transport
Trace: Parquet ZSTD (128MB groups)

What GNS3 adds

  • Visual topology + lifecycle control
  • Internal virtual switching (bridges/OVS)
  • Shareable lab projects

What NetLab produces

  • transactions.parquet (golden stimulus)
  • run.json manifest + logs
  • Optional semantic traces (RTEdbg)

What NetLab avoids

  • Always-on waveform capture
  • Immediate dependency on emulation
  • Full cycle-accurate signoff
Tick/Tack packaging: Start with tick (two visible nodes: CPU + SIM) for modular development. When stable, package as tack (one wrapper node) without changing external ports or debug entrypoints.

Tick vs Tack (how it appears in GNS3)

Same ports + same debug entrypoints; different packaging
Guideline: develop and evolve functionality in tick. When interfaces and operations stabilize, ship tack to simplify consumption.

How to use (operator workflow)

Fast reproduction → capture trace → replay downstream
  1. Build topology in GNS3: management switch + dataplane switch(es) + nodes (CPU, SIM, clients).
  2. Start SIM node first (Debug Portal gRPC server, trace writer ready).
  3. Start CPU node, run the probe tool to exercise MMIO + MSI-X init path.
  4. Capture trace via Debug Portal (StartTrace / StopTrace) → transactions.parquet.
  5. Escalate only if needed: replay the same trace in DebugLab-Lite / DebugLab-Full.
# Example (conceptual)
TraceService.StartTrace(output_dir="run_001/",
  trace_buses=["PCIE_MMIO","MSIX"],
  parquet_compression="ZSTD",
  parquet_row_group_size_mb=128)

# Run probe/regression in tick-switch-cpu...

TraceService.StopTrace()
Best practice: treat NetLab traces as “portable bug capsules”. A single captured run should be enough to drive deeper waveform-level debugging later.

DebugLab-Lite

The waveform-capable, open-source debug environment. It replays the same golden stimulus captured earlier (transactions.parquet) into a Verilator RTL backend, producing FST waveforms (GTKWave-friendly) and correlating them with RTEdbg semantic traces.

Waveforms: FST
Replay: stimulus-only / stimulus+check
Trace: Parquet ZSTD (128MB groups)
Control: Debug Portal (gRPC)
Semantics: RTEdbg (+ optional VCD markers)
Inputs
  • run.json manifest
  • transactions.parquet golden stimulus
  • Optional semantic trace configuration
Outputs
  • waves.fst waveforms
  • Observed transactions.parquet (cycle tb)
  • RTEdbg decoded reports (CSV/text)
Key mechanism: corr_id is globally unique per run and anchors transaction traces, semantic logs, and waveform windows. This makes it practical to capture only a minimal region around first_bad_corr_id.

How to use (operator workflow)

Typical commands are driven via gRPC clients/CLI
  1. Load the golden stimulus trace.
  2. Replay (start with DMA ignored):
ReplayService.LoadStimulus(stimulus_uri=".../transactions.parquet",
  mode=STIMULUS_PLUS_CHECK,
  timing_policy=IGNORE_TIMING,
  ignore_buses=["PCIE_DMA"])

ReplayService.RunReplay()
  1. When mismatch occurs, fetch first_bad_corr_id from replay results.
  2. Capture a minimal waveform window around that corr_id:
WaveformService.StartCapture(
  window_corr_id_start=N-50,
  window_corr_id_end=N+50,
  format="FST",
  signal_profile="pcie")
Tip: Keep the default waveform profile small. Expand signal sets only after you’ve localized the failure region.

DebugLab-Full

Target: PCIe-attached device/ASIC + x86_64 Intel/AMD management CPU.

DebugLab-Full is the high-fidelity environment that escalates through a structured model ladder: SystemC LT/AT → IA → CA → RTL simulation → acceleration → emulation. The core workflow is replay + compare: replay the same golden stimulus, compare traces, and capture only a minimal waveform window around first_bad_corr_id when emulation is unavoidable.

Vendor note: DebugLab-Full is architected to be EDA-vendor agnostic. Commercial tool integration is deployment-specific; Cadence, Synopsys, and Siemens stacks are supported as examples below.
Common stimulus: transactions.parquet
Common control: Debug Portal (gRPC)
Compare key: seq + corr_id
Waveforms: backend-native exports
Goal: minimize emulation usage

What it reuses

  • NetLab evidence bundles
  • Same Parquet schema v1
  • Same corr_id semantics

What it adds

  • IA/CA + deeper RTL simulation
  • Acceleration options
  • Native waveform databases (tool-specific)

When to use emulation

  • Only after simulation cannot resolve the bug
  • Only with a narrowed corr_id window
  • Only with trace-compare context attached
Operational rule: every stage exports a comparable transactions.parquet. Emulation is a targeted “surgical capture”, driven by trace-compare results.
11

Fidelity ladder + evidence loop (corr_id driven)

Capture windows are driven by corr_id: replay into the next engine, export observed Parquet, compare to find first_bad_corr_id, then capture the smallest possible waveform window around that region.

Fidelity ladder (conceptual)

  • SystemC LT/AT — fast reproduction + ordering/timing relationships
  • Instruction Accurate (IA) — ISA correctness for embedded cores
  • Cycle Accurate (CA) — timing-critical correctness for key blocks
  • RTL Simulation — signal-level debug + waveforms
  • Acceleration (optional) — higher throughput than pure RTL sim
  • Emulation (last resort) — minimal capture window using first_bad_corr_id
Outcome: emulation captures are smaller, faster, and more targeted; capacity is reserved for cases that truly require it.
12

How to use (escalation playbook)

Same gRPC calls across engines. Start with the lowest-cost engine that can reproduce the issue, then escalate.

  1. Start with a NetLab CEF bundle and pick the next backend (AT → IA → CA → RTL sim).
  2. Load stimulus (v1 starts with DMA ignored):
ReplayService.LoadStimulus(stimulus_uri=".../transactions.parquet",
  mode=STIMULUS_PLUS_CHECK,
  timing_policy=IGNORE_TIMING,
  ignore_buses=["PCIE_DMA"])

ReplayService.RunReplay()
  1. Compare observed traces to find divergence: first_bad_corr_id.
  2. If still unresolved, run emulation and capture only the corr_id window.
Best practice: treat wave capture as a targeted investigation tool. Use trace-compare as your “radar” to find where to look.

Vendor-specific integrations (deployment-dependent)

This section maps vendor tool stacks onto the ladder. The architecture does not depend on the vendor; it depends on: stimulus (transactions.parquet), correlation (seq + corr_id), uniform control (Debug Portal gRPC), and minimal capture around first_bad_corr_id.

Cadence stack (example)

  • Stratus HLS: derive RTL from SystemC/C++; support SystemC-based flows.
  • Virtual System Platform (VSP) + tlmgen: automate SystemC TLM-2.0 peripheral/IP model creation.
  • Xcelium: RTL simulation and emulation interaction hub.
  • Helium Platform: virtual platform enablement and early software workflows.

Synopsys stack (example)

  • Platform Architect: early architecture analysis/optimization (SystemC/TLM-style).
  • Virtualizer: build and deploy virtual prototypes.
  • VCS: RTL + SystemC mixed simulation/verification flows.

Siemens stack (example)

  • Veloce: emulation / prototyping / SW bring-up.
  • Questa: RTL simulation / verification.
  • Catapult: SystemC/C++ high-level synthesis and verification.
